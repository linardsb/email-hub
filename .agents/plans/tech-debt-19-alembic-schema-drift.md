# Tech-Debt 19 — Alembic Schema Drift Reconciliation

**Cluster:** B (highest severity — known-bug, destructive risk).
**Closes:** `tech-debt-alembic-schema-drift`.
**Branch:** `tech-debt/19-alembic-drift`.
**Estimated effort:** 1–2 sessions (one for the model sweep, one for the
no-op normalization migration + CI promotion). May fold into Session 19 backend
sweep alongside F057 migration squash if scheduled together.

## Problem

`make check-full` runs `alembic check` with `continue-on-error: true`
(`.github/workflows/ci.yml:200`). It reports drift but doesn't fail the build.
Surface area documented in `docs/migration-debt.md`. Concrete deltas:

| Class of drift | Tables affected | Fix |
|---|---|---|
| `TIMESTAMP()` (no tz) vs. model `DateTime(timezone=True)` | ~15 tables (blueprint_checkpoints, calibration_records, calibration_summaries, collaborative_documents, component_qa_results, design_connections, design_import_assets, design_imports, design_token_snapshots, esp_connections, rendering_screenshots, rendering_tests, …) | `ALTER COLUMN … TYPE TIMESTAMPTZ` in normalization migration |
| `nullable=True` in DB vs. `nullable=False` on model `Mapped[datetime]` | 5 tables × 2 cols = 10 cols (rendering_screenshots, rendering_tests, calibration_records, calibration_summaries, component_qa_results created_at + updated_at) | NULL backfill (`UPDATE … SET col = NOW() WHERE col IS NULL`) → `ALTER COLUMN … SET NOT NULL` |
| `items` table + 3 indexes still in DB but model deleted in `eddcd1ac` (F007) | `items` only | `op.drop_table("items")` + `op.drop_index(...)` |
| `qa_overrides_qa_result_id_key` unique constraint exists in DB, not on model | `qa_overrides` | Add `UniqueConstraint("qa_result_id", name="qa_overrides_qa_result_id_key")` to model |
| `ix_memory_entries_embedding_hnsw` HNSW vector index in DB, not on model | `memory_entries` | Declare in model with `Index(..., postgresql_using="hnsw", ...)` |
| Column comment drift on `design_connections`, `design_token_snapshots` | 2 tables | Reconcile model `comment=` strings vs. DB |
| PK index drift `ix_*_id` missing on a dozen tables | many | Suppress in `alembic/env.py` via `include_object` filter (Postgres auto-creates the PK btree; the explicit `ix_<table>_id` is redundant) |

If anyone runs `alembic revision --autogenerate` against this state without
reading the deferred-items entry, the autogenerated migration will be a 200+-op
mass that includes a `drop_table("items")` plus column-type rewrites — landing
that blindly is a production-corruption event.

## Approach: single normalization migration + CI promotion

Land **one** migration with all reconciliations, in this order (this matches the
`closes_when` clause of the deferred-items entry):

1. NULL backfill (`UPDATE …`) — must run before the `SET NOT NULL`s.
2. Drop stale `items` table + indexes.
3. `ALTER COLUMN` on all timestamp columns to `TIMESTAMPTZ`.
4. `SET NOT NULL` on the 10 affected created_at/updated_at columns.
5. Reconcile qa_overrides unique constraint (declare on model, no-op DDL).
6. Reconcile memory_entries HNSW index (declare on model, no-op DDL).
7. Reconcile column comments (`COMMENT ON COLUMN`).

Then update `alembic/env.py` to suppress PK-index `add_index` findings, and drop
`continue-on-error: true` from CI.

## Steps

### 1. Pre-flight: enumerate the actual drift

```bash
git checkout -b tech-debt/19-alembic-drift
make db-migrate
uv run alembic check 2>&1 | tee /tmp/alembic-drift.log
```

Capture the full check output. Cross-reference against the table above; if the
real output diverges (e.g. an extra table has drift, or one of the listed tables
no longer drifts), update this plan before continuing.

### 2. NULL audit (gate for SET NOT NULL)

The 10 NOT NULL targets (created_at + updated_at on 5 tables) need a NULL count
to decide whether the backfill is `NOW()` or "investigate why these rows
exist":

```sql
SELECT 'rendering_screenshots' AS tbl,
       SUM(CASE WHEN created_at IS NULL THEN 1 ELSE 0 END) AS null_created,
       SUM(CASE WHEN updated_at IS NULL THEN 1 ELSE 0 END) AS null_updated
  FROM rendering_screenshots
UNION ALL
SELECT 'rendering_tests',           SUM(CASE WHEN created_at IS NULL THEN 1 ELSE 0 END),
                                    SUM(CASE WHEN updated_at IS NULL THEN 1 ELSE 0 END)
  FROM rendering_tests
UNION ALL
… (calibration_records, calibration_summaries, component_qa_results)
```

Run against staging (or a snapshot). If NULL counts are >0, the migration
should backfill with `NOW()` AT TIME ZONE 'UTC' and log a warning. If the
production DB has rows with NULL `created_at`, that is itself a finding — record
in the migration's docstring before the `op.execute()` line.

### 3. Generate the normalization migration

```bash
uv run alembic revision -m "normalize_schema_drift"
```

Hand-write the body — do **not** use `--autogenerate`. The migration emits raw
DDL via `op.execute()` for the `ALTER COLUMN`s (Alembic's high-level helpers
choke on `TIMESTAMPTZ` conversions on populated tables in some configurations).

Skeleton:

```python
def upgrade() -> None:
    # 1. NULL backfill on the 5 affected tables (gate for SET NOT NULL).
    for tbl in ("rendering_screenshots", "rendering_tests", "calibration_records",
                "calibration_summaries", "component_qa_results"):
        op.execute(f"UPDATE {tbl} SET created_at = NOW() WHERE created_at IS NULL")
        op.execute(f"UPDATE {tbl} SET updated_at = NOW() WHERE updated_at IS NULL")

    # 2. Drop stale items table.
    op.drop_index("ix_items_owner_id", table_name="items", if_exists=True)
    op.drop_index("ix_items_created_at", table_name="items", if_exists=True)
    op.drop_index("ix_items_id", table_name="items", if_exists=True)
    op.drop_table("items", if_exists=True)

    # 3. ALTER COLUMN to TIMESTAMPTZ on the ~30 columns.
    timestamp_drift = [
        ("blueprint_checkpoints", ["created_at", "updated_at"]),
        ("calibration_records", ["created_at", "updated_at"]),
        ("calibration_summaries", ["created_at", "updated_at"]),
        … # full list from /tmp/alembic-drift.log
    ]
    for tbl, cols in timestamp_drift:
        for col in cols:
            op.execute(
                f"ALTER TABLE {tbl} ALTER COLUMN {col} "
                f"TYPE TIMESTAMP WITH TIME ZONE USING {col} AT TIME ZONE 'UTC'"
            )

    # 4. SET NOT NULL on the 10 created_at/updated_at columns.
    not_null_targets = [
        ("rendering_screenshots", ["created_at", "updated_at"]),
        … # 5 tables from §2 above
    ]
    for tbl, cols in not_null_targets:
        for col in cols:
            op.alter_column(tbl, col, nullable=False)

    # 5. qa_overrides unique constraint (DB exists; declare in model below
    #    in the same PR so alembic check sees parity).
    # 6. memory_entries HNSW index — same: model-only declaration, no DDL.
    # 7. Column comments.
    op.execute("COMMENT ON COLUMN design_connections.<col> IS '<text from model>'")
    …


def downgrade() -> None:
    # Reverse order. Drops are NOT reversible (items, NOT NULL → nullable, TZ → no-tz).
    # Document the irreversible parts in the migration docstring.
    raise NotImplementedError(
        "Schema drift normalization is forward-only. "
        "If rollback is required, restore from snapshot."
    )
```

### 4. Squawk gate

`make migration-lint` runs squawk against the new migration. The risky rules
are:

- `disallow-unique-on-existing-table` — the qa_overrides unique constraint is
  declared on model only (no DDL), so squawk shouldn't fire.
- `prefer-text-field` and `column-type-change` — the `TIMESTAMPTZ` conversions
  will fire. Add a `.squawk.toml` exclusion **only for this migration** via
  `# squawk-ignore: column-type-change` comments above each `op.execute()`.

If squawk flags any conversion as table-rewriting on a >10M-row table, split
that table's conversion into a separate migration that uses `BEGIN; SET
lock_timeout = ...; ALTER TABLE ...; COMMIT;` and document the operational
runbook in the migration docstring.

### 5. Model updates (parity side of the reconciliation)

For each "DB has X, model doesn't" item:

- **qa_overrides unique constraint:** add `UniqueConstraint("qa_result_id",
  name="qa_overrides_qa_result_id_key")` to the model `__table_args__`.
- **memory_entries HNSW index:** declare `Index("ix_memory_entries_embedding_hnsw",
  "embedding", postgresql_using="hnsw", postgresql_with={"m": 16,
  "ef_construction": 64}, postgresql_ops={"embedding": "vector_cosine_ops"})`.
  Confirm parameters against the actual DB index definition first
  (`SELECT indexdef FROM pg_indexes WHERE indexname =
  'ix_memory_entries_embedding_hnsw'`).
- **Column comments:** copy the exact string from the DB into the model's
  `mapped_column(..., comment="...")`.

For the "model has X, DB doesn't" PK-index direction, the fix lives in
`alembic/env.py`:

```python
def include_object(obj, name, type_, reflected, compare_to):
    # Postgres auto-creates a btree index on the PK column; alembic's
    # autogenerate emits an explicit ix_<table>_id which is redundant noise.
    if type_ == "index" and name and name.endswith("_id") \
            and name.startswith("ix_") and obj.unique is False:
        cols = [c.name for c in obj.columns]
        if cols == ["id"]:
            return False
    return True

context.configure(
    …,
    include_object=include_object,
)
```


each remaining finding is either: (a) a missed item from the table at the top

this plan was written — investigate before merging.
